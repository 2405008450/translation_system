# 项目基本框架

> 项目名称：**AI Translation System**（项目化翻译工作台 + TM/术语/词汇表资产管理）
>
> 更新时间：**2026-07-22**
>
> 技术栈：**FastAPI + SQLAlchemy 2 + PostgreSQL(pg_trgm/pgvector) + Redis/ARQ + Vue 3 + Vite + Pinia + vue-i18n + Naive UI + ECharts**
>
> 定位：一个面向内部翻译流程的 Web 工作台，支持项目创建、文件上传、多格式句段抽取、多阶段指派、TM/术语/词汇表辅助、AI 预翻译与修正、参考资料分析、术语与拼写语法 QA、批注协作、修订确认、项目合并视图、英美式英语副本、数据看板和译后文件导出。

## 3. 后端入口与配置

| 文件 | 作用 |
| --- | --- |
| `app/main.py` | 校验运行配置；调用 `ensure_runtime_schema()`；挂载 `auth`、主业务、术语库、参考资料、词汇表和工具路由；初始化导入任务存储；可调大 anyio 线程池；提供 `/api/health` 和 `/api/app-version`；生产环境托管 `frontend/dist` |
| `app/config.py` | 管理 `DATABASE_URL`、连接池、PgBouncer 事务池模式、上传限制、导入/导出目录、Redis/ARQ、LibreOffice/Aspose、TM 向量、DeepSeek/OpenRouter、LanguageTool、JWT、CORS、日志级别等 |
| `app/database.py` | 创建 SQLAlchemy engine；配置连接池、`application_name`、时区或 PgBouncer `prepare_threshold=None`；提供 `SessionLocal`、`Base`、`get_db()` |
| `app/auth.py` | `passlib[bcrypt]` 密码哈希；`python-jose` JWT；角色 `super_admin/admin/user`；译者类型 `internal/external`；项目访问权限判断；管理员依赖 |
| `app/logging.py` | 后端日志格式与级别 |
| `app/services/schema_setup.py` | 启动时检查关键表/列和历史命名；必要时补齐部分运行时 schema |

关键运行时约束：

- `JWT_SECRET_KEY` 不能保留默认值 `change-me-in-production`，否则后端拒绝启动。
- 生产推荐通过 PgBouncer 事务池连接数据库，并设置 `DATABASE_PGBOUNCER_TRANSACTION_MODE=true`。
- `REDIS_URL` 未配置时部分缓存和队列能力会退回本地模式；生产建议配置 Redis。
- AI 修正至少需要配置 `DEEPSEEK_API_KEY` 或 `OPENROUTER_API_KEY`。
- 拼写/语法 QA 依赖可选的自托管 LanguageTool：`LANGUAGETOOL_BASE_URL`。

### 4.2 主业务 API（`app/routers/api.py`）

所有主业务接口默认挂在 `/api` 下，并通过 `get_current_user` 要求登录。`/documents`、`/tm`、部分 `/termbase` 仍保留为隐藏兼容路径，新功能优先使用新命名。

| 分组 | 主要能力 / 路径 |
| --- | --- |
| 健康与版本 | `/api/health`、`/api/app-version`（在 `main.py`） |
| 数据看板 | `GET /api/analytics/dashboard?granularity=day\|month`；返回总览、趋势、语言对、来源分布、用户活跃及 LLM 模型用量 |
| 通知 | `GET /api/notifications`、`PATCH /api/notifications/read-all`、`PATCH /api/notifications/{id}/read` |
| 指派记录 | `GET /api/assignment-events`、`GET /api/projects/{project_id}/assignment-events` |
| 解析 | `POST /api/parser/slate`、`POST /api/parser/workspace`、`POST /api/parser/parse`、`GET /api/parser/formats` |
| 文件上传能力 | `GET /api/file-records/upload-capabilities` |
| 项目 | `GET/POST /api/projects`、`GET/PATCH/DELETE /api/projects/{id}`、`POST /api/projects/{id}/duplicate`、`POST /api/projects/{id}/source-document` |
| 项目工作流 | `GET /api/workflow-templates`、文件工作流 transition preview / transition |
| 项目指派 | `GET/PATCH /api/projects/{project_id}/assignments`、`PATCH /api/file-records/{file_record_id}/assignment`；项目指派支持按文件整篇授权或通过 `file_ranges` 指定 1-based 句段闭区间 |
| 项目资源与质量设置 | `GET/PATCH /api/projects/{project_id}/translation-memory-settings`、`term-base-settings`、`quality-qa-settings`；质量设置当前落地拼写/语法检查，其余规则保留配置位 |
| 项目文件 ZIP 导出 | `POST /api/projects/{project_id}/file-export-zip-tasks`、查询和下载导出任务 |
| 项目合并视图 | `GET/POST /api/projects/{project_id}/merge-views`、`GET/PATCH/DELETE /api/merge-views/{view_id}`、合并视图句段/QA/术语报告接口 |
| 任务文件 | `POST /api/file-records`、`GET /api/file-records`、`GET/DELETE /api/file-records/{id}`、`POST /api/file-records/{id}/duplicate` |
| 英文变体副本 | `POST /api/projects/{project_id}/file-records/{file_record_id}/english-variant-copy`；在中文→美式/英式英语任务之间自动判断方向并创建待复核副本；`british-english-copy` 为旧兼容接口 |
| 文件操作锁 | `POST/PATCH/DELETE /api/file-records/{id}/operation-lock`，避免导入、导出、预翻译等操作互相踩踏 |
| 句段分页 | `GET /api/file-records/{id}/segments`，支持分页、范围、状态、源文/译文搜索和模糊搜索，并返回文件工作流阶段进度 |
| 句段编辑 | `PUT /api/file-records/{id}/segments/{sentence_id}`、`PUT /api/file-records/{id}/segments`、源文编辑、拆分、合并、确认状态、全文替换；批量保存响应会回传最新状态统计和阶段进度 |
| 项目句段同步 | 文件级/项目级 `project-sync` 开关，支持同源句段异步同步；同步结果使用独立 `project_sync` 来源/统计桶，不再混入 TM 精确匹配 |
| 预览与定位 | `GET /api/file-records/{id}/preview`、句段 position 查询、changes 查询 |
| TM 候选与重匹配 | `GET /api/file-records/{id}/segments/{segment_ref}/tm-candidates`、`POST /api/file-records/{id}/rematch`、`PATCH /api/file-records/{id}/bindings` |
| 保存到 TM | `GET /api/file-records/{id}/save-to-tm/stats`、`POST /api/file-records/{id}/save-to-tm` |
| 预翻译任务 | `POST /api/projects/{project_id}/pretranslation-runs`、`GET /api/pretranslation-runs/{run_id}`、active 查询、取消 |
| AI 修正 | `POST /api/file-records/{id}/llm-translate`，SSE 事件包含 start / segment / error / complete |
| 翻译规则 | `GET/POST/PUT/DELETE /api/guideline-templates*`，模板存入 `prompt_templates/translation_guidelines/` |

| 文档统计 | `POST /api/projects/{project_id}/document-statistics`、报告列表和详情 |

`beautifulsoup4`（导入名 `bs4`）的当前依赖边界：

- 仅 `app/services/quote_converter.py` 使用，具体位于 HTML/HTM 引号转换分支 `_convert_html()`。
- `BeautifulSoup` 用于解析 HTML 并遍历文本节点，`Comment` 用于跳过注释，`NavigableString` 用于安全替换节点；转换弯引号时还会创建带字体样式的 `<span>`。
- 主任务 HTML 解析适配器和 `language_detection.py` 均不依赖 `bs4`；前者使用适配器自身逻辑，后者使用标准库 `HTMLParser`。
- 因此 `requirements.txt` 中的 `beautifulsoup4>=4.12,<5.0` 是引号转换工具的直接运行依赖，不是 FastAPI、参考资料分析或通用文件解析的基础依赖。
- 当前 HTML 实现按文本节点分别调用引号状态机；若一对引号横跨 `<b>`、`<span>` 等标签，后半个引号可能被重新判定为左引号。这是合入代码的已知测试缺口，不是 `bs4` 依赖本身的问题。

文档统计口径补充：

- DOCX 统计默认采用纯 OpenXML 解析，不启动 LibreOffice；统计正文和表格正文中的可见文本。
- 该口径会排除文本框、删除修订、隐藏文本、页眉页脚等非主体内容，避免把不应计入 Word-like 正文字数的内容混入统计。
- 统计结果新增结构数量字段：图片数、去重图片数、内嵌图片数、浮动图片数、外链图片数、图表数、SmartArt 数。
- `.doc` 文件仍先通过 LibreOffice 转为 `.docx`，再进入同一套 OpenXML 统计流程；转换或解析失败时才使用兜底统计或返回不可用。

## 7. 多格式解析与导出

适配器系统位于 `app/services/adapters/`，围绕 `FormatAdapter`、`AdapterRegistry`、`DocumentAST`、`Segment`、`ParseResult` 工作。`__init__.py` 当前默认注册：

```text
txt, docx, xlsx, pdf, pptx, dita, svg, yaml, json, php, html,
properties, po, strings, markdown, srt, csv, sdlxliff, txml,
dxf, zip, idml, mif, rar
```

`quote_converter.py` 虽然同样处理多种文件格式，但属于 `/api/tools` 下的独立字节转换工具，不经过上述任务适配器注册表，也不会生成项目句段。

当前任务上传入口实际开放格式以 `task_file_service.get_supported_task_extensions()` 和前端 `src/constants/taskFiles.ts` 为准：

```text
.doc, .docx
.pptx, .xlsx
.txt, .dat, .csv
.html, .htm, .md, .markdown
.json, .yaml, .yml, .php, .properties
.po, .pot, .strings, .srt
.dita, .ditamap, .xml, .svg
.sdlxliff, .txml
.dxf, .idml, .mif
.zip
```

说明：

- 代码里有 `pdf_adapter.py` 和 `rar_adapter.py`，但当前上传白名单没有开放 `.pdf` / `.rar`。
- `.doc` 上传会通过 LibreOffice 转为 `.docx` 后进入 Word 解析流程。
- `.pptx`、`.xlsx` 当前已开放上传，并支持原格式写回导出。
- `.zip` 会递归解析包内支持格式，按内部文件回写导出。
- DITA/XML 的“内联元素不拆分”以 `<p>` 等文本块为边界；`<tgroup>/<tbody>` 或专用化产生的未知结构容器会递归保留其后代块级节点，不能把整张 `<table>` 聚合成一个句段。
- 导出选项包括 `original`、`bilingual`、`bilingual_docx`、`bilingual_docx_layout_source_first`、`bilingual_docx_layout_target_first`、`bilingual_txt`、`tmx`、`xliff`、`xliff2`，不同扩展名支持集由 `export_formats.py` 控制。

### 7.1 Word 句段换行与版式还原

DOCX 解析、预翻译、编辑和导出对换行采用“双文本轨道”：

- `source_text` 继续保存规范化文本，用于 `source_hash`、TM 精确/模糊匹配、搜索、去重和历史兼容，不把 Word 版式换行并入匹配文本。
- `display_text` 用于工作台展示；新解析结果同时携带 `source_layout_text`，保留普通 `w:br` 和表格单元格多段落合并产生的 `\n`。
- 旧数据没有 `source_layout_text` 时，运行时可从 `display_text` 回退计算，避免引入数据库迁移。
- LLM 提示优先使用版式原文；真实换行会编码为 `⟦LB_1⟧`、`⟦LB_2⟧` 等不可翻译占位符，写回前还原为普通 `\n`。
- LLM 校验会检查原文版式换行数量，模型删除换行时触发重试或错误，避免预翻译吞掉 Word 单元格内的换行。
- 前端译文编辑统一保存普通 `\n`；`SegmentEditorRow.vue` 支持插入换行按钮、`Shift+Enter` 和 `Ctrl/Cmd+Enter`。
- DOCX 导出不能把 `\n` 直接写入 `<w:t>`；普通软换行写成 `<w:br/>`，表格单元格原本跨多个 `<w:p>` 的场景优先按译文行分配回原段落，多余行克隆最后一个段落样式。

## 9. 数据库

推荐 PostgreSQL 14+，生产镜像使用 `pgvector/pgvector:pg17`。必须启用：

```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE EXTENSION IF NOT EXISTS vector;
```

### 9.1 初始化与迁移

新库推荐执行：

```powershell
psql -U postgres -d tm_demo -f scripts/init_db.sql
```

Docker 生产环境由 `docker-compose.prod.yml` 完成：

- `postgres` 首次启动挂载 `scripts/init_db.sql`、`scripts/create_reference_profiles.sql` 等初始化脚本。
- `db-migrate` 每次启动执行 `scripts/run_migrations.sh`，按文件名顺序运行 `scripts/migrations/*.sql`。
- 当前幂等迁移链已覆盖 `0001`—`0021`，主要包括：句段 trigram 与稳定顺序索引、项目合并视图、修订显示设置、分段指派范围、翻译规则、资源创建者、TMX 元数据、pgvector、Auto-TM、TM 检索投影、参考资料同步、句段元数据、文件级句段统计、项目同步 outbox、TM 检索范围、TM/术语库持久化条目数及 LLM 模型看板索引。
- 目录中目前同时存在两组 `0005` 和两组 `0021` 文件；部署脚本按完整文件名排序，并在每次启动时重新执行全部幂等脚本，因此两组文件都会独立执行。新增迁移不得仅凭数字前缀判断顺序，也必须保持可重复执行。

历史库升级时仍可能需要按现场状态执行 `scripts/add_*.sql`、`scripts/create_*.sql`、`scripts/rename_translation_memory_tables.sql`、`scripts/migrate_primary_keys_to_uuid.sql` 等补丁。

### 9.2 关键索引与性能约定

- `memory_entries.source_hash`：精确匹配。
- `memory_entries.source_normalized`：归一化精确和模糊匹配。
- `memory_entries.source_text/source_normalized`：GIN + `gin_trgm_ops` 模糊召回。
- `memory_entries.source_embedding`：pgvector 语义召回，配合 `source_embedding_version`。
- `memory_entry_search`：面向大规模 TM 的轻量检索投影；触发器仅在原文、语言对、所属库或 hash 等检索字段变化时重建投影，降低写放大。
- `memory_bases.entry_count`、`term_bases.entry_count`：语句级触发器维护资源条目数，避免资源列表对大表实时 `COUNT + GROUP BY`。
- `segments(file_record_id, display_index)` 与源结构/`sequence_index`：工作台稳定分页和原格式导出顺序。
- `segments.source_text/display_text/target_text`：trigram 搜索。
- `segments(updated_at, llm_model) WHERE source='llm'`：看板按时间和模型聚合 MT 句段。
- `file_segment_stats`：以触发器增量维护文件状态统计，确认、轮询和进度计算不再重复全文件聚合。
- `project_segment_sync_outbox(status, last_enqueued_at)`：项目重复句段同步 worker 的待办扫描。
- `segment_revisions(file_record_id/segment_id/status)`、`segment_comments(file_record_id/segment_id/parent_id)`：工作台协作加载。
- `project_assignments`、`file_assignments`、`assignment_events`、`notifications`：项目权限和通知查询；`file_assignments.segment_range_start/end` 用于外部译者分段可见/可写判断。

### 11.1 创建项目并上传源文档

  ├─ file_record_service.attach_source_document_to_file_record

### 11.2 工作台分页编辑与自动保存

```text
WorkbenchView / segment store
  │ GET /api/file-records/{id}
  │ GET /api/file-records/{id}/segments?skip=0&limit=100
  │ GET /api/file-records/{id}/preview?skip=0&limit=100
  │ GET /api/file-records/{id}/revisions
  │ GET /api/file-records/{id}/comments
  ▼
用户编辑句段
  ├─ 前端 dirtyEntries 暂存
  ├─ 防抖后 PUT /api/file-records/{id}/segments
  ├─ file_record_service.batch_update_segments
  └─ revision_service.create_revision 生成修改快照
```

外部译者的工作台句段读取与写入都由后端按 `workflow_step_id + display_index` 计算。读取时，后续流程阶段的译者可查看当前阶段及之前阶段的句段，以便获得完整审阅上下文；写入仍要求句段当前阶段与译者被指派阶段完全一致。若 `FileAssignment.segment_range_start/end` 为空，则范围覆盖整文件；若填写范围，则接口只下发落在该 1-based 闭区间内的句段。单句保存、批量保存、确认、替换、拆分、合并和项目同步都会复用同一套范围校验。

派稿到非首流程阶段时，后端会检查本次新增授权范围内是否仍有句段停留在紧邻的上一阶段。若有，首次保存返回结构化预检结果，前端要求操作者选择“指派并进入目标阶段”或“仅指派，暂不流转”。前者在同一数据库事务中完成授权写入、句段阶段推进和状态重置；后者只保存授权，译者可只读查看前序内容。系统不会自动跨越两个及以上流程阶段。

句段分页、增量 changes 和批量保存接口都会返回 `workflow_progress`。`segment.ts` 将其写回当前文件或合并视图文件，工作台底栏据活动句段的 `workflow_step_id` 显示当前阶段进度；项目页则按所有阶段汇总总流程进度。百分比最多保留两位小数，当前阶段同时显示 `completed_segments/total_segments`。

### 11.3 项目级预翻译

```text
ProjectDetailView / PreTranslateDialog
  │ POST /api/projects/{project_id}/pretranslation-runs
  ▼
pretranslation_runs + pretranslation_tasks
  ├─ 可选择 TM 重匹配、LLM 修正、术语/词汇表绑定
  ├─ local 后台任务或 Redis + ARQ worker
  ├─ 操作锁保护文件
  ├─ 通过 notifications / active task 查询进度
  └─ 任务完成后刷新项目进度和工作台句段
```

预翻译 LLM 阶段的稳定性约定：

- `PreTranslateDialog.vue` 提交 `llm_translation_unit` 时默认使用 `sentence`。句段模式不依赖模型返回段落级严格 JSON，能避免某一组 `sentence_id/source_hash` 校验失败导致整组句段全部失败。
- 后端仍支持 `paragraph`，但它属于上下文增强模式：会按段落/表格单元格组装上下文并要求模型返回严格 JSON，适合确认模型能稳定遵守 `translations`、`sentence_id`、`source_hash` 合同后再启用。
- Word 句段如带版式换行，LLM 阶段使用 `source_layout_text` 并把换行显式标记为 `⟦LB_n⟧`；结果写回前还原为 `\n`，校验失败会触发重试或记录错误。
- 后台 ARQ 队列按职责隔离：`arq:import` 处理上传和资源导入，`arq:auto-tm` 处理 auto-TM outbox 和自动重匹配，`arq:segment-sync` 处理项目重复句段同步，`arq:maintenance` 保留给手动 QA 和遗留维护任务，`arq:pretranslation` 处理预翻译 run。
- 预翻译队列分两层限流：`ARQ_PRETRANSLATION_MAX_JOBS` 控制同时执行多少个 run，`PRETRANSLATION_RUN_FILE_CONCURRENCY` 控制单个 run 内同时处理多少个文件；每个文件的 LLM 阶段还受 `LLM_MAX_CONCURRENCY` 限制。
- 预翻译任务即使整体状态为 `completed`，只要 `error_segments > 0`，前端也必须展示 `pretranslation_tasks.error`。不要只展示“成功/失败计数”，否则会掩盖真实原因，例如 JSON 校验失败、符号/数字保护失败或数据库写回失败。

- 大规模 TM/TB：检索投影、持久化条目计数、文件句段统计与项目同步 outbox。
- 术语导入、词汇表导入、术语抽取、术语 QA。
- LLM：provider fallback、规则模板、临时提示词、符号/数学占位符/版式换行保护、语言对传递。
- 英美式英语转换：词典/规则转换、HTML 保真、双向副本、文件名去重和语言对限制。
- 项目状态、项目复制、任务删除级联、项目文件 ZIP 导出。
- 文件操作锁、导入任务存储、异步导出队列。
- 项目/文件指派：外部译者可见性、同文件多人分段可见/可写权限、空范围兼容和范围重叠校验。
- 文档统计、重复统计、匹配分析。

当前测试缺口：`quote_converter.py` 与 `/api/tools/quote-convert` 尚无专门的后端自动化测试，尤其需要补充 HTML 节点保留、编码回退、Office 格式保真和 50 MB 上传边界用例。

## 14. 关键约定

1. **接口命名**：新开发优先使用 `/api/projects`、`/api/file-records`、`/api/translation-memory`、`/api/term-bases`、`/api/glossary-bases`、`/api/reference`；`/api/documents`、`/api/tm`、部分 `/api/termbase` 属于兼容路径。
2. **语言对**：项目、任务、TM、术语库、词汇表都要保持源语言/目标语言一致；后端通过 `language_pairs.py` 规范化别名。
3. **角色权限**：`super_admin` / `admin` 可管理用户；管理员和内部译者均可管理项目、指派、项目设置、资源库及统计等业务功能；外部译者默认只进入分配给自己的任务，并按文件指派的流程阶段与句段范围获得逐段可见/写权限。新增用户及修改用户表单信息仅限管理员。
4. **解析模式**：Word 支持 `full` 和 `body_only`，并带页眉页脚、脚注尾注、批注、超链接保留等解析选项。
5. **任务文件白名单**：前后端开放格式以 `frontend/src/constants/taskFiles.ts` 和 `task_file_service.get_supported_task_extensions()` 为准。
6. **大文档策略**：工作台句段、预览、修订、批注、搜索、替换都优先走服务端分页或后端全文处理，避免前端加载全文。
7. **句段状态与来源**：状态包括 `exact`、`fuzzy`、`none`、`confirmed`；`source` 记录 `tm`、`manual`、`llm`、`project_sync`、`english_variant_conversion` 等来源。看板和状态统计将 `project_sync` 独立成桶，不再计作 TM 精确匹配。
8. **修订机制**：人工编辑和 LLM 写回都会创建 `segment_revisions`，前端可接受或拒绝。
9. **批注机制**：批注支持句段级和选区级锚点，回复通过 `parent_id` 形成线程。
10. **操作锁**：长耗时操作应先持有 file operation lock，完成或失败后释放。
11. **TM 匹配策略**：精确匹配只认 `source_hash`；模糊匹配混合 trigram、字符串相似度和可选 pgvector 得分。
12. **AI 输出保护**：LLM 结果会校验数字片段、复选框/特殊符号、数学公式占位符和版式换行，失败会通过 SSE `error` 返回。
13. **Word 换行保留**：`source_text` 只用于匹配和 hash，不承载版式换行；Word 解析保留 `source_layout_text`，译文换行统一保存为 `\n`，导出 DOCX 时再转换成 `<w:br/>` 或表格单元格 `<w:p>` 结构。
14. **预翻译 LLM 默认粒度**：批量预翻译默认传 `llm_translation_unit="sentence"`；`paragraph` 只作为可选上下文增强模式使用，避免模型未稳定遵守严格 JSON 合同时出现整组句段失败。
15. **预翻译错误可见性**：`pretranslation_tasks.error` 不只在任务 `failed` 时展示；当 `error_segments > 0` 时也要在前端状态中展示，便于直接定位模型输出校验、写回数据库或锁状态问题。
16. **翻译规则来源**：可复用规则使用 UTF-8 Markdown；用户在工作台或预翻译中手动输入的内容只作为本次临时提示词。
17. **资源类型区分**：TM 用于句段级记忆匹配；术语库用于译后术语一致性检查；词汇表用于 AI 预翻译上下文注入。
18. **TM 检索范围**：`selected` 只检索选中的库；`language_pair_all` 按语言对覆盖所有库并避免超长 ID 列表。无论哪种范围，都必须先校验任务与资源语言对一致。
19. **项目同步链路**：确认后的同源句段通过 `project_segment_sync_outbox` 异步传播；同步译文保留来源句段/文件 ID，`PROJECT_SYNC_CONFIRMED_ONLY` 控制是否只同步已确认内容。
20. **阶段进度口径**：项目页显示全流程汇总进度，工作台底栏显示活动句段所属阶段进度；接口写回后必须同步刷新 `workflow_progress`，不能只在首次加载文件时计算。
21. **导入导出队列**：大文件导入、资源导出、文件导出优先走任务状态接口；生产建议使用 Redis + ARQ worker。
22. **安全配置**：`.env` 不入库；`JWT_SECRET_KEY` 必须替换；LLM API Key、数据库密码等只放环境变量。
23. **中文编码**：代码和文档按 UTF-8 处理，中文注释、错误提示、前端文案和 Markdown 文档统一避免乱码。
24. **独立工具边界**：`/api/tools/quote-convert` 是无数据库写入的即时文件转换接口；其多格式处理不属于项目任务 adapters，`bs4` 也只服务于该工具的 HTML 分支。
