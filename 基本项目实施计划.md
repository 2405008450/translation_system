# AI 翻译系统核心功能技术落地与架构方案

本方案针对当前项目的核心目标进行收敛：先以现有 `Vue 3` 生态为主完成可落地的翻译工作台，同时保留后续演进到 `React` 富文本生态的技术路线，但不在当前阶段为了潜在需求提前切栈。

## 1. 当前架构定案

### 1.1 前端技术栈定案

*   **主前端继续使用 Vue 3**：基于当前项目已经存在 Vue 3 生态，后续工作台、任务管理、术语管理、审校流转等业务都继续落在 Vue 3 上实现。
*   **当前不整体切换到 React**：现阶段的核心目标是“结构化文档工作台”，不是“浏览器内完整 Word 编辑器”。在该目标下，Vue 3 可以实现与 React 接近的交互效果。
*   **保留 React 升级路线**：如果后续某个子模块演化成真正的重富文本编辑器，可将该子模块单独拆分为 React 应用或 React 编辑器内核，而不是整站重构。

### 1.2 文档解析与展示路线定案

*   **DOCX 主解析引擎采用 `python-docx`**：用于提取段落、表格、单元格、文本节点等结构化信息，并建立后端统一文档模型。
*   **系统内部统一维护 `Document AST + Segment` 模型**：
    *   `Document AST` 负责表达文档结构，如段落、表格、单元格、文本块。
    *   `Segment` 负责表达最小翻译单元，如句子、短语或后续更细粒度片段。
*   **右侧正文区域优先采用“只读预览 + 高亮定位”模式**：当前阶段不做浏览器内完整富文本编辑器，不追求直接在正文区域中任意改写原文结构。
*   **当前正文渲染优先使用结构化 HTML 方案**：后端输出带锚点的 HTML 或 JSON 结构，前端负责渲染、高亮、跳转和状态联动。
*   **`Mammoth` 作为可选增强路线**：如果未来需要更高的 Word 样式保真度，可以引入 `Mammoth` 生成 HTML 作为展示层补充，但不作为核心数据模型来源。

### 1.3 断句、锚点与精确定位定案

*   **断句后的每个翻译单元都必须有稳定 ID**：不能仅使用一次请求内的随机 ID，而应基于文档版本、块路径、顺序或持久化策略生成稳定 `sentence_id / segment_id`。
*   **定位精度定义为“字符级/片段级”**：当前需求是对句子、短语、局部文本进行精确修改和重渲染，不涉及字节级处理。
*   **允许后续从句级扩展到更细粒度片段**：例如对句内术语、数字、变量占位符、时间日期等局部内容单独识别和替换。

### 1.4 匹配、LLM 与人工审校定案

*   **翻译流程采用分层策略**：
    1. 精确匹配
    2. 标准化精确匹配
    3. 模糊匹配
    4. 术语库提示
    5. LLM/MT 建议生成
    6. 人工审校确认
*   **LLM 不直接覆盖最终结果**：LLM 输出默认作为建议译文，需标记来源和状态，由人工确认后才能进入最终结果或回写 TM。
*   **LLM 调用采用异步任务方式**：避免整篇文档同步阻塞，优先针对 `none` 或低分 `fuzzy` 片段做增量建议生成。
*   **人工修订后的高质量译文可回写 TM**：但需要经过状态控制，仅在满足审核条件时写入翻译记忆库。

### 1.5 多格式支持定案

*   **V1 以 DOCX/TXT 为主**：先把 Word 文档的解析、断句、匹配、审校、回写流程打通。
*   **XLSX 当前阶段主要用于 TM 导入**：不强行纳入统一审校工作台。
*   **PDF/PPTX 暂不纳入 V1 主流程**：仅在架构层面预留“格式适配器”接口。
*   **后续扩展采用“适配器模式”而非一个前端控件统一吃掉所有格式**：
    *   `DOCX Adapter`
    *   `XLSX Adapter`
    *   `PDF Adapter`
    *   `PPTX Adapter`

## 2. 业务实现方式定案

### 2.1 核心工作流

1.  用户上传原始文档。
2.  后端创建文档记录与版本记录。
3.  解析引擎将文档转换为 `Document AST`，同时抽取 `Segment`。
4.  每个 `Segment` 进入 TM 检索流程，得到 `exact / fuzzy / none` 等状态。
5.  对未命中或低置信度片段，异步调用 LLM 生成建议译文。
6.  前端工作台展示“左侧译文编辑区 + 右侧原文预览区”，实现定位、高亮、状态联动。
7.  人工审校确认后，更新最终译文、状态、备注、来源信息。
8.  审核通过的译文按规则回写 TM，并用于后续导出或版本沉淀。

### 2.2 前端交互边界

*   **左侧为主编辑区**：用户主要在左侧修改译文、接受/拒绝建议、查看匹配来源、触发 LLM 改写。
*   **右侧为原文结构预览区**：用于快速定位、高亮、跳转、比对，不作为当前阶段的自由富文本主编辑区。
*   **允许局部实时重渲染**：当某个片段被人工或 LLM 修改后，前端可局部更新状态和预览，不必整页刷新。
*   **后续可扩展为片段内增强操作**：如术语替换、数值修正、风格改写、上下文重写、批量接受建议。

## 3. 为什么当前不切 React

### 3.1 当前需求还不等于“超重富文本编辑器”

当前已明确的需求主要是：

*   解析文档结构
*   按句或片段进行定位
*   对译文做字符级修改
*   由 LLM 提供建议并重新渲染
*   左右区域联动高亮与跳转

这些需求本质上属于“结构化文档工作台”，并不等于浏览器内 Word/Google Docs 级别的编辑器。

### 3.2 什么情况下才值得单独引入 React 富文本生态

当后续出现以下能力时，再评估单独引入 `React + Slate/Lexical`：

*   用户直接在正文主区域进行自由编辑，而不是只在左侧编辑结果字段
*   需要跨段落、跨单元格、跨块级节点的复杂选区操作
*   需要撤销重做、复杂快捷键、拖拽重组、批注、修订痕迹、多光标协作
*   需要在浏览器中承载接近 Word 的复杂编辑事务模型

如果只是“句级/片段级编辑 + 原文预览联动 + LLM 建议”，Vue 3 完全可以承担。

## 4. 为未来 React 路线预留的架构约束

为了避免未来真的切换或增加 React 子模块时成本过高，当前架构必须遵守以下约束：

*   **前后端通信只使用稳定 JSON 协议**：不能把前端框架特定结构直接写入接口协议。
*   **文档模型独立于前端框架**：`Document AST`、`Segment`、匹配结果、建议结果都由后端定义。
*   **工作台状态机独立于页面组件**：如匹配状态、审校状态、LLM 状态、回写状态不能耦合在某个 Vue 组件内部。
*   **编辑器模块可独立拆分**：如果未来真要引入 React，应只替换“编辑器子模块”，而不是推翻整套业务系统。
*   **优先使用可跨生态迁移的编辑器理念**：例如以结构化模型和事务接口为核心，而不是过早把业务逻辑深度绑定到某个前端库的渲染细节。

## 5. 调整后的实施计划

### 第一阶段：打通 DOCX 工作台主链路

1.  重构 DOCX 解析能力，输出统一 `Document AST + Segment`。
2.  为每个片段建立稳定 `segment_id`，支持前后端定位。
3.  接入现有 TM 检索，形成 `exact / fuzzy / none` 状态结果。
4.  完成 Vue 3 工作台基础交互：左侧列表、右侧预览、双向高亮、定位跳转。

### 第二阶段：补齐译文生产与审校闭环

1.  增加建议译文、人工修订、状态流转、来源标记。
2.  引入异步 LLM 建议生成能力。
3.  增加术语提示、局部改写、批量处理等操作。
4.  建立审核通过后回写 TM 的规则和数据流。

### 第三阶段：扩展版本化与更复杂业务

1.  引入文档版本、审校记录、任务记录。
2.  支持多角色协作，如译员、审校员、管理员。
3.  规划导出能力与最终交付格式。
4.  根据实际需求评估是否需要引入独立 React 富文本子模块。

## 6. 结论

当前方案正式定为：

*   **主系统继续使用 Vue 3**
*   **DOCX 解析核心采用 `python-docx`**
*   **正文采用“结构化预览 + 精确定位”，而不是当前阶段直接建设浏览器内完整 Word 编辑器**
*   **LLM 采用异步建议模式，不直接覆盖人工结果**
*   **保留未来引入 React 富文本子模块的技术路线，但不在当前阶段整体切栈**

这样可以在现阶段更快落地，同时保留后续架构上限，不把未来可能的编辑器升级成本提前转化为当前项目的整体复杂度。
