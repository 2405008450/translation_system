# 云服务器部署常用指令

项目目录示例：`~/opt/translation_system`

## 首次部署

```bash
cd ~/opt/translation_system
git pull
cp .env.prod.example .env.prod
nano .env.prod   # 改密码、JWT、LLM Key、CORS 公网 IP

chmod +x scripts/deploy_prod.sh

# 标准：默认启 nginx，对外发布 80
scripts/deploy_prod.sh up

# 需要 Mihomo 代理时
USE_PROXY=1 scripts/deploy_prod.sh up
```

安全组放行 **TCP 80**。

## 代码更新 / 重新构建

```bash
cd ~/opt/translation_system
git pull

echo "=== 当前版本 ==="
git log -1 --oneline

# 重建 app 镜像并滚动 app + 各类 worker
scripts/deploy_prod.sh restart

# 查看状态与日志
scripts/deploy_prod.sh ps
```

## 并发参数第一档

```env
ARQ_IMPORT_MAX_JOBS=2
ARQ_MAINTENANCE_MAX_JOBS=2
ARQ_AUTO_TM_MAX_JOBS=1
ARQ_SEGMENT_SYNC_MAX_JOBS=1
ARQ_PRETRANSLATION_MAX_JOBS=2
```

预翻译理论请求上限约为 `ARQ_PRETRANSLATION_MAX_JOBS × PRETRANSLATION_RUN_FILE_CONCURRENCY × LLM_MAX_CONCURRENCY`。如果模型接口出现 429、超时或错误率升高，先把 `LLM_MAX_CONCURRENCY` 降回 2；如果服务器仍很轻松且模型接口稳定，再逐步提高。

```bash
WITH picked AS (
  SELECT collection_id, source_language, target_language, source_normalized
  FROM memory_entries
  WHERE source_normalized IS NOT NULL
    AND collection_id IS NOT NULL
    AND source_language IS NOT NULL
    AND target_language IS NOT NULL
```

把上一步输出的 `collection_id / source_language / target_language / source_normalized` 填入下面命令：

```bash
sudo docker exec -i ai-translation-postgres psql -U tm_user -d tm_demo -P pager=off <<'SQL'
SET pg_trgm.similarity_threshold = 0.3;
EXPLAIN
SELECT tm.id
FROM memory_entries AS tm
WHERE tm.source_normalized IS NOT NULL
  AND tm.source_normalized % '这里填 source_normalized 样本文本'
  AND tm.collection_id = '这里填 collection_id'::uuid
  AND tm.source_language = '这里填 source_language'
  AND tm.target_language = '这里填 target_language'
```

## memory_entries 分区迁移规划

低峰维护窗口不是“随便找个晚上跑命令”，而是提前约定一段用户很少的时间，例如凌晨 01:00-04:00。维护窗口内应暂停大导入、批量预翻译、auto-TM、资源库重算等重任务，只保留必要前台访问，先做快照/备份，再执行迁移，最后验证和恢复 worker。

直接把 `memory_entries` 改成按语言对分区并不属于小改，原因：

- 当前表约千万级记录、数十 GB，迁移会大量读写磁盘并重建索引。
- PostgreSQL 原生分区表的全局唯一约束有限制，主键/唯一约束通常需要包含分区键；现有 `id` 是单列 UUID 主键，直接改成语言对分区会影响约束设计。
- 迁移期间还要处理新写入的 TM、auto-TM outbox、导入任务和回滚。

推荐更稳的两步走：

1. 先保留 `memory_entries` 作为权威主表。
2. 新建一个专门给 fuzzy 检索用的 `memory_entry_search` 投影表，按 `source_language + target_language` 分区，必要时对子分区按 `collection_id` hash。
3. 投影表只存检索必要字段：`entry_id`、`collection_id`、`source_language`、`target_language`、`source_normalized`、`updated_at`。
4. fuzzy 先查分区投影表拿 `entry_id`，再回 `memory_entries` 取完整 `source_text / target_text / creator` 等信息。
5. 用批处理回填历史数据，用触发器或应用写入逻辑保持增量同步。
6. 验证稳定后，再考虑是否迁移主表本身。

如果确实要完整迁移主表，大致步骤如下：

```text
1. 维护窗口前：
   - 确认磁盘剩余空间至少能容纳新表 + 新索引 + 回滚旧表。
   - 创建数据库快照或 pg_dump 备份。
   - 暂停 auto-tm-worker、导入 worker、预翻译 worker。

2. 建新表：
   - 创建 memory_entries_new，字段与旧表一致。
   - PARTITION BY LIST (source_language, target_language)。
   - 对 zh-CN/en-US、en-US/zh-CN 等大语言对建立独立分区。
   - 大分区可继续按 collection_id HASH 子分区。

3. 建索引：
   - 每个分区建立 collection + language + source_hash 的 exact 索引。
   - 每个分区建立 source_normalized gin_trgm_ops 索引。
   - 建唯一约束时必须重新评估是否包含分区键。

4. 回填数据：
   - 按 collection_id 或 id 范围分批 INSERT INTO memory_entries_new SELECT ...
   - 每批后记录进度，避免一次事务过大。

5. 增量同步：
   - 维护窗口短时停写，或在旧表上建触发器把新增/更新同步到新表。

6. 切换：
   - 应用停写。
   - 校验行数、重复约束、抽样查询结果。
   - ALTER TABLE memory_entries RENAME TO memory_entries_old;
   - ALTER TABLE memory_entries_new RENAME TO memory_entries;
   - 重建必要权限、触发器、外键、统计信息。

7. 验证：
   - 跑 EXPLAIN，确认 partition pruning 和分区索引命中。
   - 跑预翻译、右侧匹配信息、导入、保存到 TM 的冒烟测试。
   - 恢复 worker。

8. 回滚：
   - 如验证失败，停应用写入。
   - 把新表改名，memory_entries_old 改回 memory_entries。
   - 恢复服务。
```

更推荐优先执行“检索投影表分区”方案，因为它不需要马上改动权威主表和主键约束，风险比完整主表分区低很多。

## memory_entry_search 检索投影表

这个方案保留 `memory_entries` 作为权威主表，只新增一张轻量投影表 `memory_entry_search` 给 TM fuzzy 检索使用。投影表只保存检索必要字段：`entry_id`、`collection_id`、语言方向、`source_hash`、`source_normalized`、`source_length`、`updated_at`。当前实现按 `source_length` 分区，因为 fuzzy 查询本身带长度范围，能够先剪掉明显不相关的长度分区，再用 trigram 索引搜索。

它能改善的点：
- 前台 TM 预翻译、右侧“匹配信息”、候选列表不再直接在 37GB 宽表上做最重的 `% / similarity`。
- 主表只在候选命中后按 `entry_id` 回表读取 `source_text / target_text / creator` 等完整信息。
- 新增、更新、删除 TM 时由数据库触发器同步投影表，避免遗漏某条写入链路。
- 默认 `TM_SEARCH_PROJECTION_ENABLED=false`，回填完成前不会改变线上查询路径。

它不能替代的点：
- 不是搜索引擎，也不是向量库；如果某个单独记忆库本身继续膨胀到千万级，后续仍可能需要更细的 collection 分片、归档或专用检索服务。
- 回填期间会增加磁盘写入和索引体积，所以仍建议低峰慢速执行。

创建投影表和触发器：

```bash
sudo docker-compose --env-file .env.prod -f docker-compose.prod.yml -f docker-compose.proxy.yml run --rm db-migrate
```

先试跑 2 个小批次，确认没有锁等待和异常：

```bash
sudo docker exec -it ai-translation-app python scripts/backfill_memory_entry_search.py --batch-size 10000 --limit-batches 2 --sleep 0.5
```

正式低速回填：

```bash
sudo docker exec -it ai-translation-app python scripts/backfill_memory_entry_search.py --batch-size 20000 --sleep 0.2
```

回填完成后刷新统计信息：

```bash
sudo docker exec -i ai-translation-postgres psql -U tm_user -d tm_demo -c "ANALYZE memory_entry_search;"
```

如果中断，记录上一行日志里的 `last_id`，继续执行：

```bash
sudo docker exec -it ai-translation-app python scripts/backfill_memory_entry_search.py --batch-size 20000 --sleep 0.2 --resume-from 这里填_last_id
```

查看投影表覆盖量和分区大小：

```bash
sudo docker exec -i ai-translation-postgres psql -U tm_user -d tm_demo -P pager=off <<'SQL'
SELECT
  (SELECT count(*) FROM memory_entries
   WHERE source_normalized IS NOT NULL
     AND char_length(source_normalized) > 0
     AND source_language IS NOT NULL
     AND target_language IS NOT NULL) AS source_rows,
  (SELECT count(*) FROM memory_entry_search) AS projection_rows;

SELECT
  relid::regclass AS partition_name,
```
